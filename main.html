1 Introduction: Why Kanban?

Kanban (Japanese for "signboard") is a visual workflow management method designed to help us visualize work, maximize efficiency, and improve continuously.

Unlike time-boxed sprints (Scrum), our Kanban process emphasizes continuous flow. This allows us to adapt priorities flexibly, reduce context switching, and spot bottlenecks immediately.

2 Our 7-Stage Workflow

Our board is divided into 7 distinct stages to ensure quality and control. Each stage has clear entry criteria, ownership, and goals.

Stage 1 Ideas / Inputs
PM Vision / Stakeholders

Raw ideas, feature requests, and bug reports. No development activity happens here. This is our capture zone to ensure nothing is lost.

üéØ Goal: Capture everything so nothing is lost
Stage 2 Backlog
Product Owner (PO)

Approved ideas waiting for prioritization and refinement.

Entry Criteria:
  • Clear title and description
  • Business value defined
  • Approved by PO
Stage 3 Ready (Quality Gate)
Team

Fully defined features ready for development. This is our commitment point.

Definition of Ready (DoR):
  • Requirement is unambiguous
  • Acceptance criteria defined
  • Effort estimated (points/hours)
  • No external blockers
Stage 4 In Progress
Developer

Active development phase. Work is pulled from "Ready" only when capacity exists.

Rule: One owner per card. Strict WIP (Work In Progress) limits apply. Never push work‚Äîonly pull when ready.
Stage 5 Review / QA
QA / Peers

Validation against acceptance criteria and quality standards.

Activities:
  • Code review
  • Unit testing
  • Design compliance
  • Stakeholder demo
Critical Rule: If issues are found, do not move the card backward. Keep it here and create sub-tasks for fixes.
Stage 6 Blocked
Team / Scrum Master

Work halted due to external impediments (e.g., waiting on API, dependency on another team).

Action Required: Must include a reason tag. The team "swarms" to resolve these immediately.
Stage 7 Done
Team

Completed value delivered to customers.

Definition of Done (DoD):
  • Code complete & merged
  • All tests passing
  • Code review approved
  • Documentation updated

3 Practical Scenarios & Workflow Scripts

Use these scripts to handle common situations during the software development lifecycle.

The "Happy Path" Feature
Standard flow from concept to completion
[Ideas: Feature Request] ‚Üí (PO Approval) ‚Üí [Backlog] ‚Üí (Refinement & Sizing) ‚Üí [Ready] ‚Üí (Pull Signal) ‚Üí [In Progress] ‚Üí (Code Complete) ‚Üí [Review/QA] ‚Üí (DoD Met) ‚Üí [Done]
  1. Ideas/Inputs: A stakeholder adds a card: "Add 'Dark Mode' to User Profile." It sits here as a raw concept.
  2. Gate to Backlog: The Product Owner reviews the idea. It aligns with the roadmap, so the PO approves it, adds priority, and moves it to Backlog.
  3. Backlog to Ready: During grooming, the Team reviews the card. They define acceptance criteria ("Switch toggles UI colors"), estimate effort, and confirm no dependencies. The card moves to Ready.
  4. Ready to In Progress: A Developer finishes their current task. Checking Ready, they pull the top-priority "Dark Mode" card to In Progress.
  5. In Progress to Review: Developer completes code and unit tests. They move the card to Review/QA.
  6. Review to Done: A Peer/QA validates the feature. It passes all criteria. The card meets the Definition of Done and is moved to Done.

The "QA Kickback" (Defect Found)
Handling defects during Review stage
Key Rule: Never move cards backward on the board. Flow only moves right!
[Review/QA: Bug Found] ‚Üí (STOP: Do Not Move Back) ‚Üí (Create Sub-task/Fix) ‚Üí (Verify Fix) ‚Üí [Done]
  1. Context: The "User Login" feature is in Review/QA.
  2. Discovery: The QA Engineer finds the "Forgot Password" link is broken.
  3. Critical Decision: The card remains in Review/QA to maintain accurate flow metrics.
  4. Action: QA flags the card (Red Label). A sub-task is added: "Fix broken password link."
  5. Resolution: The Developer pauses other work to fix this immediately (swarming).
  6. Completion: Once the fix is verified, the card is moved to Done.

The "External Blocker"
Dependency on external teams
[In Progress] ‚Üí (External Dependency Hit) ‚Üí [Blocked] ‚Üí (Owner Assigned) ‚Üí (Dependency Resolved) ‚Üí [In Progress]
  1. In Progress: A Developer is building an API integration.
  2. The Roadblock: They need a secure API Key from the "Security Team" (external). Work halts.
  3. Move to Blocked: The Developer moves the card to Blocked.
  4. Documentation: They write on the card: Reason: Waiting for API Key from Security Team.
  5. Ownership: The Scrum Master takes ownership of following up with Security.
  6. Resolution: The key is provided. The card is moved back to In Progress.

The "WIP Gridlock" (Board Full)
Handling WIP limit constraints
Key Rule: Stop starting, start finishing!
[Ready: Full] ‚Üí [In Progress: Full] ‚Üí [Review/QA: Full] ‚Üí (Developer Free) ‚Üí (DECISION: Do Not Pull New) ‚Üí (Swarm on Review) ‚Üí [Done]
  1. Context: In Progress is at its WIP limit (5/5). Review/QA is also full (3/3).
  2. The Situation: A Developer finishes a task and wants new work.
  3. The Restriction: They cannot pull from Ready because In Progress is full.
  4. The Swarm: Instead of violating the limit, the Developer looks at Review/QA.
  5. Action: They pick up a teammate's pending code review.
  6. Result: The review is completed, moving that card to Done. Space is created, and flow resumes.

The "Vague Request" (Gatekeeping)
Rejecting unclear requirements
[Ideas: Vague Input] ‚Üí (Backlog Attempt) ‚Üí (REJECTED: Entry Criteria Fail) ‚Üí [Ideas: Updated] ‚Üí [Backlog]
  1. Ideas/Inputs: Stakeholder adds: "Make the app faster."
  2. Review: Product Owner reviews the card.
  3. Rejection: The card fails Definition of Ready for Backlog (No specific scope).
  4. Feedback: PO comments: "Please define 'faster'‚Äîis this page load time? Which screen?"
  5. Refinement: Stakeholder updates: "Reduce Home Screen load time to < 2 seconds."
  6. Approval: PO moves the card to Backlog.
